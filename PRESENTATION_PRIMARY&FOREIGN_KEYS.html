Why would we use relational databases? How do they serve us in the long run?

So the first DBMS (database management system) was created in 1960, and was basically just a single file that contained a list of every single record contained in a given database. So in order to find something, a computer had to search through everything in the list line-by-line, from beginning to end, which was very tedious, resource-consuming, and slow. In the early 70s, Edgar F. Codd wrote a breakthrough paper while working at IBM detailing the relational model, which inspired IBM to develop their then quasi-relational DBMS into the beginnings of SQL (which was actually called SEQUEL for Structured English Query Language before having to shorten the name due to copyright infringement). Eventually ORACLE Corporation developed SQL into its first commercially available implementation in 1979, and by 1986 it was officially recognized as a standard language by ANSI and ISO.

Why the history lesson? Because its confusing to speak of database management systems (DBMS), relational database management systems (RDBMS), and non-relational database management systems (NRDBMS) without this context. It might sound like NRDBMS were first and that RDBMS evolved out of them, when in reality NRDBMS (in the commonly used sense) evolved after RDBMS, themselves evolving out of vanilla DBMS.

To reiterate:

Why would we use relational databases? How do they serve us in the long run?

By making sure all of your data are created within schemas (or schemata if you wanna sound smart) that meaningfully connect to each other, you ensure that your data maintains referential integrity. The industry term for this tegridy is ACID: Atomicity, Consistency, Isolation, and Durability. These reduce to very similar principles, so Ill just give the (wikipedia) definition of the most informative one, Atomicity, which describes an atomistic transaction as an indivisible and irreducible series of database operations such that either all occur, or nothing occurs (each series of database operations equals one transaction). Wikipedias wordy definition boils down to: each of your data (or DATUMS if it helps) has an identity that is tied to a larger whole (that larger whole being the database schema).

So, your data is

       easier to find/

       can more intricately be searched for (queried),

       can have very complex data analysis performed on it,

       and is more strictly maintained within a rigid (pre-defined) structure.

^structure here means the same thing as schema^

This is important for databases created for the purposes of accounting (transactional databases) where you depend on an always-accurate state of your database and which doesnt ever veer from its pre-defined schema (ACID-compliant). So your records are more singular and independent (Atomicity), more well-defined (Consistency), more permanent/reliably maintained (Durability), and give you a higher overall degree of control over their visibility (Isolation). In essence, ACID compliance aims to guarantee that every finalized transaction is valid even in the event of network errors, disruptions, hardware failures, etc. and is ideal for

       Banks/retail/military

       Handling time-sensitive data (e.g. transportation records/schedules)

       Monitoring sensitive data in general

       Whenever accuracy is crucial to your operations

 

What type of programming paradigm is SQL or MySQL?

SQL is of the declarative programming language paradigm (so wordy amirite? it only gets worse), where the language expresses what it is going to do, as opposed to an imperative programming language, which states how it is going to do it. It also has elements of a procedural language paradigm (which is just a type of declarative programming language characterized by its step-by-step programming style).

SQL databases == relational databases (RDBMS); they are one and the same. The logic of such databases is built on relational algebra.

Another wikipedia quote:

Relational algebra, first created by Edgar F. Codd while at IBM {remember Edgar?}, is a family of algebras with a well-founded semantics used for modelling the data stored in relational databases, and defining queries on it.

Without going into the math whatsoever, SQL/relational algebra treats data, their tables, and their relations like numbers, in order to perform (algebraic) operations on them, which are actually just fancy search queries.

In case you werent confused yet, SQL isnt even a programming language, its really just a collection of sublanguages. From Wikipedia once more:

Originally based upon relational algebra and tuple relational calculus, SQL [as a language] consists of many types of statements,[8] which may be informally classed as sublanguages, commonly: a data query language (DQL),[a] a data definition language (DDL),[b] a data control language (DCL), and a data manipulation language (DML).[c][9] Thus, the scope of SQL includes:

      data query (DQL),

      data manipulation (DML) (insert, update and delete),

      data definition (DDL) (schema creation and modification), and

       data access control (DCL).
